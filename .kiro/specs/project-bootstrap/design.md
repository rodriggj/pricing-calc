# Design Document: Project Bootstrap

## Overview

This is the design for the **Project Bootstrap** Glue spec (per `requirements.md` "Spec Shape Declaration"). The spec wires the empty workspace to the already-approved `.kiro/specs/estimate-format-and-contract/` spec by producing a working Next.js + TypeScript + pnpm + Docker Postgres + shadcn-initialized skeleton.

The design is small by intent. It is mostly:

- A pinned-version table so two developers running this spec a year apart get byte-identical lockfiles.
- The exact non-interactive CLI invocations (`pnpm create next-app`, `pnpm dlx shadcn init`) that produce the skeleton.
- The exact contents (or content patterns) for the config files this spec owns.
- The seam between this spec and the contract spec — what we explicitly do **not** create here.

There is no application logic to design. There is no API surface. There are no data models in the value-domain sense. The "design" is the configuration. Everything below serves either auditability of FR-1 through FR-5 / NFR-1 / NFR-2 or reproducibility of the demo script in `requirements.md`.

## Architecture

### Bootstrap Command Sequence

The skeleton is produced by exactly two non-interactive commands, run in order from the empty repo root, plus a small set of file edits. Each command's flag set is fixed by this design (see Components and Interfaces below).

```
1. pnpm create next-app@<pinned> . <flags>     # FR-1, FR-2 (most of)
2. <hand-edit package.json>                     # FR-2 (packageManager + engines + pin tightening)
3. <write .nvmrc>                               # FR-2
4. <write docker-compose.yml>                   # FR-3
5. <write .env.local.example>                   # FR-4
6. <update .gitignore>                          # FR-4
7. pnpm dlx shadcn@<pinned> init <flags>        # FR-5
```

Step 1 produces ~95% of the file tree. Steps 2–6 are small additive edits. Step 7 adds two files (`components.json`, `src/lib/utils.ts`) and updates `package.json` with `clsx` + `tailwind-merge`.

### Boundary with `estimate-format-and-contract`

NFR-2 in `requirements.md` is the contract between this spec and the next one. This design honors it by:

- Not adding `drizzle-orm`, `drizzle-kit`, `zod`, `fast-check`, or `vitest` to `package.json` dependencies. (Verified by Confirm step "a" in the demo.)
- Not creating any of the directories or config files listed above as **MUST NOT EXIST**. (Verified by Confirm step "e" in the demo.)
- Not adding `out/` or `migrations/` to `.gitignore` here — the contract spec's Task 1 owns those entries because it owns the directories that need them.

## Components and Interfaces

The "components" of a Glue spec are configuration files. Each is specified below by exact contents (where small) or by content-pattern + non-default flags (where the file is large and machine-generated).

### 1. Pinned Versions

The Versioning & Pinning section of `tech-stack.md` requires pinned minor versions. As of authoring (May 2026), the recommended pins are:

| Package      | Pin       | Rationale                                                                 |
| ------------ | --------- | ------------------------------------------------------------------------- |
| `next`       | `15.5.9`  | Latest fully-patched stable on the Next.js 15.5 line. Bumped from `15.5.0` → `15.5.7` to address [CVE-2025-66478](https://nextjs.org/blog/CVE-2025-66478) (RSC RCE, CVSS 10.0), then bumped again to `15.5.9` for [CVE-2025-55183 / CVE-2025-55184 / CVE-2025-67779](https://nextjs.org/blog/security-update-2025-12-11) (RSC source-code exposure and DoS). 15.5.9 is the fully patched release on the 15.5.x line; App Router GA, Turbopack stable for `next dev`. |
| `eslint-config-next` | `15.5.9` | Tracks the `next` minor and patch line.                            |
| `react`      | `19.0.0`  | Required by Next.js 15.x; React 19 is the GA line.                        |
| `react-dom`  | `19.0.0`  | Must match `react`.                                                       |
| `typescript` | `5.7.3`   | Stable TS 5.7; aligned with what `create-next-app@15` defaults to.        |
| `clsx`       | `2.1.1`   | Used by the hand-authored `cn` helper. Pinned exactly. (Previously suggested at this version; confirmed at install.) |
| `tailwind-merge` | `3.6.0` | Used by the hand-authored `cn` helper. Pinned exactly. Bumped from the original `2.5.5` suggestion to the current major (3.x has full Tailwind v4 support). |
| `tailwindcss` | (create-next-app default minor; pin tightened to exact at install time) | create-next-app picks the matching Tailwind for Next.js 15. |

Implementation phase will **tighten** the dependency strings emitted by `create-next-app` from `^15.x.x` etc. to exact `15.5.0` etc. before `pnpm install` is run, so `pnpm-lock.yaml` reflects the pins (FR-2 AC-4).

> **Open decision for user review:** the specific patch versions above are recommendations as of authoring. If the user prefers the absolute latest patch at implementation time, the implementer will substitute and the design table will be updated to match. The non-negotiable property is "exact version, no caret, no tilde, no `latest`."

### 2. Node and pnpm Version Decisions

`requirements.md` AC FR-2/2 said "Node 20.x at the time of authoring" with a flagged caveat about Node 22 going LTS in October 2024. Recommendation:

- **Node:** Pin to **Node 22.x** (active LTS as of May 2026; Node 20 is in maintenance with EOL April 2026).
  - `engines.node` value: `">=22.0.0 <23.0.0"`
  - `.nvmrc` value: `22.14.0` (exact LTS release; the implementer will substitute the latest 22.x LTS at install time and update the design table to match).
- **pnpm:** Pin to **`pnpm@10.5.0`** in the `packageManager` field.
  - Rationale: pnpm 10 is the current stable major, released January 2025; pnpm 9 is in maintenance. 10.5.0 is the latest stable on the 10 line as of authoring.

> **Both version decisions are flagged for user override.** The user can substitute Node 20.x and/or pnpm 9.x.x. The non-negotiable properties are "active LTS line for Node, current stable major for pnpm, exact version pinned in both files."

### 3. `pnpm create next-app` invocation (FR-1)

Exact command:

```
pnpm create next-app@15.5.9 . \
  --typescript \
  --tailwind \
  --eslint \
  --app \
  --src-dir \
  --import-alias "@/*" \
  --use-pnpm \
  --turbopack \
  --skip-install
```

Flag-by-flag rationale:

| Flag                        | Effect                                                                 | Why                                          |
| --------------------------- | ---------------------------------------------------------------------- | -------------------------------------------- |
| `.`                         | Generate into the current directory                                    | We are bootstrapping the existing repo root. |
| `--typescript`              | TS scaffold, emits `tsconfig.json`                                     | FR-1 AC-2; `tech-stack.md` (Language: TS).   |
| `--tailwind`                | Includes Tailwind config + `globals.css`                               | shadcn requires Tailwind; matches storyboard.|
| `--eslint`                  | Adds the Next.js ESLint config                                         | Standard hygiene; no dispute.                |
| `--app`                     | App Router (not Pages Router)                                          | FR-1 AC-1; `tech-stack.md` (App Router).     |
| `--src-dir`                 | Places `app/` under `src/`                                             | FR-1 AC-1; matches contract spec's `src/contract/` and `src/db/schema/` layout assumption. |
| `--import-alias "@/*"`      | Sets `@/*` → `src/*` in `tsconfig.json` `paths`                        | FR-1 AC-2; FR-5 AC-1 (shadcn alias).         |
| `--use-pnpm`                | Tells the bootstrapper to use pnpm                                     | FR-2 AC-1.                                   |
| `--turbopack`               | Sets `next dev --turbopack` as the default `dev` script                | Faster local iteration; Turbopack `next dev` is stable in 15.x. Deliberate choice; production builds (`pnpm build`) still use webpack so we don't trade build correctness for dev speed. |
| `--skip-install`            | Don't auto-install yet                                                 | We hand-edit `package.json` to tighten pins **before** `pnpm install` runs, so the lockfile reflects the pinned versions on first install. |

The non-interactive flag set above covers every prompt `create-next-app` would otherwise ask. This is the determinism requirement of Risk #1 below.

### 7. `docker-compose.yml` (FR-3)

Exact contents:

```yaml
services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: estimating_app
      POSTGRES_PASSWORD: dev_only_password
      POSTGRES_DB: estimating_app
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U estimating_app -d estimating_app"]
      interval: 5s
      timeout: 5s
      retries: 6

volumes:
  postgres_data:
```

Design notes:

- **No top-level `version:` key.** Compose v2 (the `docker compose` plugin, not the legacy `docker-compose` Python tool) deprecated and ignores `version:`. Including it would emit a noisy warning. Compose v2 is assumed (Risk #3 below).
- **`postgres:16-alpine` is intentional.** Smaller image, faster pulls. `docs/postgres-db-strategy.md` notes the future swap to `pgvector/pgvector:pg16` when the document-parsing spec lands; that is out of scope here.
- **Healthcheck on `pg_isready`.** Six retries × 5-second interval = 30 seconds of total grace, matching FR-3 AC-6's "healthy within 30 seconds." `pg_isready` is shipped with the postgres image, so no additional dependency.
- **Named volume.** `postgres_data` survives `docker compose down` but not `docker compose down -v`, which matches Observe (e) in the demo script.
- **No `depends_on`.** Single-service compose; nothing depends on the database from inside the compose file in v1.

### 9. `.gitignore` reconciliation (FR-4 AC-2)

`pnpm create next-app` emits a `.gitignore` whose env section already includes:

```
# local env files
.env*.local
```

(That pattern matches `.env.local`, `.env.development.local`, `.env.production.local`, etc., satisfying FR-4 AC-2 via the `.env.*.local` glob — but note `.env.local` is also matched.)

This spec's edits:

- **Add explicit entries** for clarity and grep-ability: append a small block to the `.gitignore`:
  ```
  # Project-Bootstrap explicit env-file entries (matches the create-next-app default;
  # listed explicitly so reviewers can confirm without cross-referencing globs)
  .env.local
  .env.*.local

  # Explicitly NOT ignored: .env.local.example (committed env template, FR-4 AC-1)
  !.env.local.example
  ```
- **Do NOT add** `out/`, `fixtures/`, `migrations/`, or `coverage/`. Those belong to downstream specs (NFR-2 AC-3).

The duplicate listing is harmless — git accepts repeated patterns — and the `!.env.local.example` negation is defensive. If a future broader pattern (`*.local`) is ever added, the negation prevents the example file from being accidentally ignored.

### 10. shadcn/ui initialization (FR-5) — hand-authored

**Approach:** Hand-author the four artifacts that `shadcn init` would produce (`components.json`, `src/lib/utils.ts`, `clsx` dep, `tailwind-merge` dep). Do **not** run `pnpm dlx shadcn@<x> init`.

**Pin history (recorded for auditability — three failed CLI attempts before this approach was adopted):**

1. **`shadcn@2.3.0` with `--base-color neutral --css-variables --yes`** — failed: `--base-color` and `--css-variables` are 1.x CLI flags. The 2.x+ CLIs use `--defaults` for the same outcome.
2. **`shadcn@2.3.0` with `--defaults --yes`** — failed: `shadcn@2.3.0` predates Tailwind v4 by ~6 months. Its preflight checks for a `tailwind.config.{js,ts,mjs}` file that `create-next-app@15.5.9` no longer emits when targeting Tailwind v4 (config now lives in CSS via `@import "tailwindcss"` + `@theme`). CLI bails before writing anything.
3. **`shadcn@4.7.0` with `--defaults --yes`** — failed (boundary violation): the 4.x `init` command auto-scaffolds `src/components/ui/button.tsx` and pulls in `@base-ui/react`, `class-variance-authority`, `tw-animate-css`, `lucide-react`, and `shadcn` itself as runtime deps. Directly violates FR-5 AC-3 ("no component-specific dependencies beyond those needed by `cn` itself") and the spec's `src/components/ui/` MUST-NOT-EXIST boundary.

**Why hand-authoring is the right call:** The shadcn CLI's `init` behavior has drifted significantly across majors. The 1.x flag set, the 2.x preflight, and the 4.x auto-scaffolding all moved targets in different ways. What FR-5 actually wants — `components.json`, `src/lib/utils.ts`, `clsx`, `tailwind-merge`, no components — is a small, stable set of artifacts. Hand-authoring them produces deterministic output across pnpm/Node/Tailwind upgrades, eliminates the "shadcn CLI is unpinned at run time" risk (Risk #2 below), and meets the FR-5 ACs exactly without the over-installation that any current `init` invocation produces. When the first UI-feature spec needs an actual component, that spec can run `pnpm dlx shadcn@<pinned-then-current> add <name>` against the already-correct `components.json` — `add` does not have the boundary problems `init` does.

**Artifacts to create (this is the entire FR-5 deliverable):**

1. `components.json` at the repo root, contents per §11 below.
2. `src/lib/utils.ts`, contents per §12 below.
3. `clsx` dependency in `package.json`, exact pin per §1 (`2.1.1`).
4. `tailwind-merge` dependency in `package.json`, exact pin per §1 (`3.6.0` — bumped from the design's original `2.5.5` to the current major for full Tailwind v4 support).

After writing the four artifacts, run `pnpm install` (verify exit 0) and `pnpm build` (verify exit 0) to confirm the cn helper compiles cleanly through Next.js + Tailwind v4.

**No CLI invocation is made by this spec.** The `shadcn` CLI is not added to `package.json` or `node_modules`; it remains a `pnpm dlx` tool a future spec can invoke when adding the first component.

### 11. `components.json` (FR-5 AC-1)

Hand-authored at the repo root with the following exact contents (matches what `shadcn init --defaults` would emit on the v3-era CLI, before the 4.x `init` started auto-scaffolding components):

```jsonc
{
  "style": "new-york",
  "rsc": true,
  "tsx": true,
  "tailwind": {
    "config": "",
    "css": "src/app/globals.css",
    "baseColor": "neutral",
    "cssVariables": true,
    "prefix": ""
  },
  "aliases": {
    "components": "@/components",
    "utils": "@/lib/utils",
    "ui": "@/components/ui",
    "lib": "@/lib",
    "hooks": "@/hooks"
  },
  "iconLibrary": "lucide"
}
```

**Notes:**
- The `tailwind.config` field is an empty string because Tailwind v4 has no `tailwind.config.{js,ts,mjs}` file — config lives in `src/app/globals.css` via `@import "tailwindcss"` + `@theme`. The empty string is the v4-correct value and is what the v4-aware shadcn CLI also emits.
- The `$schema` URL key is omitted; current shadcn schema-validation tooling treats it as optional. If a future shadcn `add` invocation needs it, the field can be added at that time.
- The `aliases` block uses the exact keys downstream specs (and a future `pnpm dlx shadcn add <component>`) require.

**FR-5 AC-1 verification:** App Router (`rsc: true`, `tsx: true`), `src/` layout (alias paths point at `@/components`, `@/lib/utils`, `@/components/ui`, `@/lib`, `@/hooks` — `@/*` resolves to `src/*` per `tsconfig.json`), `@/*` alias config — all satisfied.

### 12. `src/lib/utils.ts` (FR-5 AC-2)

`shadcn init` writes the standard `cn` helper:

```ts
import { type ClassValue, clsx } from "clsx";
import { twMerge } from "tailwind-merge";

export function cn(...inputs: ClassValue[]) {
  return twMerge(clsx(inputs));
}
```

This file is the entire shadcn-init runtime contribution to `src/`. No edits.

## Data Models

This spec has no value-domain data models. The Postgres database started by `docker-compose.yml` is empty until the contract spec's Task 3 applies its first migration. The `estimates`, `lineItems`, `architectureRevisions`, `shareUrlRevisions`, and `estimateAuditLog` tables are owned by `.kiro/specs/estimate-format-and-contract/` (its Tasks 2 and 3) — explicitly **not** by this spec (NFR-2).

The "data" produced by this spec is configuration, addressed in Components and Interfaces above.

## Risk Callouts

These are the things most likely to bite the implementer. Each is mitigated by a specific design choice above.

1. **`pnpm create next-app` is interactive by default.** Running it without all flags will halt waiting for keyboard input, breaking determinism and any future CI invocation. Mitigation: §3 specifies the full flag set, including `--use-pnpm`, `--app`, and `--src-dir` — none of those have default-on behavior; all must be explicit.

2. **`shadcn init` is replaced by hand-authoring the four artifacts.** Three CLI attempts (`shadcn@2.3.0` with 1.x flags, `shadcn@2.3.0` with `--defaults`, `shadcn@4.7.0` with `--defaults`) all failed for different reasons: stale flags, Tailwind-v3-only preflight, and 4.x auto-scaffolding components in violation of FR-5 AC-3. The CLI's behavior has drifted significantly across majors. Mitigation: §10 specifies the four artifacts to hand-author (`components.json`, `src/lib/utils.ts`, `clsx` dep, `tailwind-merge` dep), with no CLI invocation. The shadcn CLI is not added to `package.json` or `node_modules`; a future UI-feature spec invokes `pnpm dlx shadcn@<pinned> add <name>` against the already-correct `components.json` when an actual component is needed.

3. **Compose v1 (the hyphenated `docker-compose` Python tool) is end-of-life.** This design assumes the v2 plugin (`docker compose`, no hyphen). The healthcheck syntax and the absent `version:` key both require v2. If a developer's machine only has v1 installed, `docker compose up -d` will print "command not found" and the demo fails fast. Mitigation: the prerequisites line in `requirements.md` already lists "Docker (or Docker Desktop) installed" — Docker Desktop ≥ 2020 ships v2 by default. On Linux, the installer instructions for `docker-compose-plugin` are distro-specific and listed in Docker's own docs. We do not attempt to detect v1 vs v2 in code.

## Error Handling

Failure modes during scaffolding and how they manifest:

| Failure                                                       | Where it surfaces                              | Recovery                                                       |
| ------------------------------------------------------------- | ---------------------------------------------- | -------------------------------------------------------------- |
| Wrong Node version (<22 or >=23)                              | `pnpm install` warns; `next dev` may run anyway. | Switch to Node 22 via `nvm use` (the `.nvmrc` is the cue).     |
| Wrong pnpm version                                            | Corepack (if enabled) auto-installs the pinned pnpm; otherwise `pnpm install` warns. | Run `corepack enable` and re-run; or install pnpm@10.5.0 directly. |
| Port 5432 already in use (e.g., a host Postgres is running)   | `docker compose up -d` exits non-zero with "address already in use." | Stop the host Postgres or change the published port in `docker-compose.yml` (the spec does not parameterize this — change is local to the developer). |
| Docker daemon not running                                     | `docker compose` exits with "Cannot connect to the Docker daemon." | Start Docker Desktop / `systemctl start docker`. |
| Network failure during `pnpm install` or `pnpm dlx shadcn`    | `pnpm install` retries with cached metadata; `dlx` cannot — it must download the CLI. | Retry when connectivity returns; the lockfile (once written) makes future installs cache-friendly. |
| `shadcn init` fails to detect the alias                       | Emits `components.json` with `aliases` pointing at the wrong path. | Inspect §11; if the file disagrees, delete it, fix `tsconfig.json` aliases, re-run init. (This is unlikely given `--src-dir` and `--import-alias` were set explicitly.) |

There is no application-level error handling because there is no application logic.

## Testing Strategy

### PBT Deferral (Re-affirmed)

`requirements.md` "Property-Based Testing — Documented Deferral" already documents the deferral under Rule 6's "invariant is trivially true by construction" allowance. **This design adds no PBT work.** No `fast-check` is installed; no Correctness Properties section is written; no property tests are run.

The reason restated for the record: this spec produces configuration files and a vendor-supplied app skeleton. The "invariants" of those artifacts are static facts (the `postgres` service uses image `X`, `.env.local` is gitignored, `pnpm dev` boots a server). There is no input space over which a property generator could meaningfully sample. PBT machinery (`fast-check`, `vitest`) is installed by `.kiro/specs/estimate-format-and-contract/` Task 1 — this spec is the precondition for that, not a participant in it.

### What this spec verifies, and how

| Aspect                                  | Verification mechanism                                              |
| --------------------------------------- | ------------------------------------------------------------------- |
| Skeleton boots                          | `pnpm dev` reaches `http://localhost:3000` (demo step 7).           |
| Skeleton builds                         | `pnpm build` exit code 0 (demo step 8).                             |
| Database is reachable                   | `docker exec ... psql -c "SELECT 1"` (demo step 6).                 |
| File contents are correct               | Manual `Confirm` steps (a–e) in the demo, comparing against §4 / §7 / §8 / §9 / §11 above. |
| Boundary with contract spec             | Confirm step "e" (absent paths); Confirm step "a" (absent deps).    |

### What this spec does **not** verify

- Concurrent request handling, response correctness, schema validation, or any value-domain behavior. Those are owned by `.kiro/specs/estimate-format-and-contract/` and the specs that follow it.
- CI pipeline. `requirements.md` "Out of Scope" reserves CI for a future spec.
- Production deployment. Out of scope per NFR-1.

## Out-of-Scope Reaffirmation

Items reserved for downstream specs (per `requirements.md` "Out of Scope") are restated here so the design is self-contained on the boundary:

- **No** Drizzle, Zod, fast-check, or Vitest dependencies, configs, or scripts.
- **No** `src/contract/`, `src/db/`, `scripts/contract-harness.ts`, `fixtures/`, `out/`.
- **No** Clerk, AWS SDKs, Terraform, Bedrock, Neon — all deferred per `aws-first-preference.md` "When to Revisit" and `tech-stack.md` Versioning & Pinning (Neon is the locked production DB but not provisioned in this spec).
- **No** individual shadcn components (`shadcn add` is not run).
- **No** `pgvector` extension (deferred to the document-parsing spec per `docs/postgres-db-strategy.md`).
- **No** CI configuration, no GitHub Actions, no preview-environment plumbing.

If a reviewer finds any of the above in the implementation, that's a NFR-1 / NFR-2 violation, not a design feature.

## Deviations from Defaults (Summary)

For auditability, every place this design departs from "what `pnpm create next-app` and `pnpm dlx shadcn init` would produce on default settings" is enumerated:

1. `--turbopack` flag passed to `create-next-app` (default in 15.x is now Turbopack-on for `dev`; we make it explicit so a future default flip doesn't silently change us).
2. `--skip-install` flag passed to `create-next-app` so we can tighten pins **before** the lockfile is written.
3. Hand-edit of `package.json` to convert `^X.Y.Z` ranges from `create-next-app` into exact `X.Y.Z` pins — required by `tech-stack.md` Versioning & Pinning.
4. Hand-edit of `package.json` to add `packageManager: "pnpm@10.5.0"` and `engines.node` constraints — `create-next-app` does not set these.
5. New file `.nvmrc` at the repo root — `create-next-app` does not emit one.
6. New file `docker-compose.yml` — entirely additive, not part of the Next.js scaffold.
7. New file `.env.local.example` — `create-next-app` does not emit one.
8. `.gitignore` augmented with explicit `.env.local` / `.env.*.local` entries (functionally redundant with the default `.env*.local` glob; added for grep-ability and as a defense against future negation patterns) and a `!.env.local.example` negation.
9. ~~`shadcn@2.3.0` pinned~~ Hand-author `components.json`, `src/lib/utils.ts`, and the `clsx` + `tailwind-merge` deps. No `pnpm dlx shadcn init` invocation. Three CLI attempts revealed that the 1.x→2.x→4.x evolution moved targets across flag set, Tailwind major support, and component auto-scaffolding behavior; hand-authoring produces stable output. Detail in §10.

No other deviations. Everything else is the upstream default.
